# Phase 11: Distribution & Shell Completions - Research

**Researched:** 2026-03-12
**Domain:** GoReleaser, Homebrew tap distribution, Cobra shell completions
**Confidence:** HIGH

## Summary

This phase packages the completed Revenium CLI for distribution via GoReleaser (cross-platform binaries on GitHub Releases) and Homebrew (macOS/Linux install via tap). Shell completions for bash/zsh/fish are already provided by Cobra's built-in `completion` command (registered in `cmd/root.go` line 107 via `SetCompletionCommandGroupID("config")`), so no new Go code is needed for the completion subcommands themselves.

The main deliverables are: (1) a `.goreleaser.yml` configuration file, (2) a `scripts/completions.sh` script to pre-generate completion files for bundling, (3) a GitHub Actions release workflow at `.github/workflows/release.yml`, and (4) documentation in the completion command's help text showing users how to source completions manually. The Homebrew formula is auto-generated by GoReleaser and pushed to the `revenium/homebrew-tap` repository.

**Primary recommendation:** Use GoReleaser v2 with the `brews` section (not `homebrew_casks`) for Homebrew formula generation. The `brews` section was briefly deprecated in v2.10 but un-deprecated in v2.12 and remains the correct choice for CLI tools because `homebrew_casks` causes `brew upgrade` issues for non-GUI applications.

## Architecture Patterns

### New Files to Create
```
.goreleaser.yml                    # GoReleaser configuration
scripts/completions.sh             # Shell completion generation script
.github/workflows/release.yml     # GitHub Actions release workflow
```

### Pattern 1: Completion File Pre-generation
**What:** A shell script that runs `go run main.go completion <shell>` for bash/zsh/fish before GoReleaser builds archives, writing output to a `completions/` directory.
**When to use:** During GoReleaser's `before.hooks` phase.
**Example:**
```bash
#!/bin/sh
# scripts/completions.sh
# Source: https://carlosbecker.com/posts/golang-completions-cobra/
set -e
rm -rf completions
mkdir completions
for sh in bash zsh fish; do
    go run main.go completion "$sh" > "completions/revenium.$sh"
done
```

### Pattern 2: GoReleaser Build Configuration with ldflags
**What:** Mirror the existing Makefile ldflags in `.goreleaser.yml` so binaries have embedded version info.
**When to use:** Always -- the `builds` section must set ldflags matching `internal/build.Version`, `internal/build.Commit`, `internal/build.Date`.
**Example:**
```yaml
# Source: GoReleaser docs + existing Makefile pattern
builds:
  - main: .
    binary: revenium
    env:
      - CGO_ENABLED=0
    ldflags:
      - -s -w
      - -X github.com/revenium/revenium-cli/internal/build.Version={{.Version}}
      - -X github.com/revenium/revenium-cli/internal/build.Commit={{.Commit}}
      - -X github.com/revenium/revenium-cli/internal/build.Date={{.Date}}
    goos:
      - linux
      - darwin
      - windows
    goarch:
      - amd64
      - arm64
```

### Pattern 3: Homebrew Formula with Bundled Completions
**What:** The `brews` section in `.goreleaser.yml` specifies how to install the binary and completion scripts via Homebrew.
**When to use:** Always -- this is how `brew install revenium/tap/revenium` works.
**Example:**
```yaml
# Source: GoReleaser docs + goreleaser's own config
brews:
  - repository:
      owner: revenium
      name: homebrew-tap
    directory: Formula
    homepage: https://github.com/revenium/revenium-cli
    description: Manage your Revenium account from the command line
    license: MIT
    install: |-
      bin.install "revenium"
      bash_completion.install "completions/revenium.bash" => "revenium"
      zsh_completion.install "completions/revenium.zsh" => "_revenium"
      fish_completion.install "completions/revenium.fish"
    test: |
      system "#{bin}/revenium", "version"
```

### Pattern 4: GitHub Actions Release Workflow
**What:** A workflow triggered on tag pushes that runs GoReleaser to build binaries, create GitHub Releases, and publish the Homebrew formula.
**When to use:** Every tagged release (e.g., `git tag v1.0.0 && git push --tags`).
**Example:**
```yaml
# .github/workflows/release.yml
name: release
on:
  push:
    tags:
      - "v*"
permissions:
  contents: write
jobs:
  goreleaser:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-go@v5
        with:
          go-version: stable
      - uses: goreleaser/goreleaser-action@v7
        with:
          distribution: goreleaser
          version: "~> v2"
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Important:** The default `GITHUB_TOKEN` cannot push to a different repository (`revenium/homebrew-tap`). The user must create a Personal Access Token (PAT) with `repo` scope and add it as a repository secret (e.g., `TAP_GITHUB_TOKEN`). The GoReleaser config should reference this token in the `brews.repository.token` field.

### Anti-Patterns to Avoid
- **Custom completion command implementation:** Cobra already provides `revenium completion bash|zsh|fish` out of the box. Do NOT create a custom `cmd/completion.go` -- it would conflict with Cobra's built-in command.
- **Using `homebrew_casks` for CLI tools:** Despite being the "newer" approach, casks break `brew upgrade` for CLI tools. Use `brews` (formulas).
- **Hardcoding version in .goreleaser.yml:** GoReleaser provides `{{.Version}}`, `{{.Commit}}`, `{{.Date}}` template variables. Never hardcode these.
- **Using `GITHUB_TOKEN` for cross-repo tap publishing:** The default token only has permissions for the current repo. Must use a PAT.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cross-platform binary builds | Custom build scripts per OS/arch | GoReleaser `builds` section | Handles CGO, cross-compilation matrix, reproducibility |
| GitHub Release creation | Manual `gh release create` | GoReleaser `release` section | Auto-generates changelog, uploads artifacts, creates release notes |
| Homebrew formula | Hand-written Ruby formula | GoReleaser `brews` section | Auto-generates formula with correct SHA256, download URLs, install steps |
| Shell completions | Custom completion logic | Cobra built-in `completion` command | Already active in the project; handles all shell syntaxes correctly |
| Release CI pipeline | Custom shell scripts | `goreleaser-action@v7` | Handles GoReleaser installation, version management, caching |

**Key insight:** This entire phase is configuration, not code. The only Go code change might be enhancing the completion command's help text. Everything else is YAML configuration files and a small shell script.

## Common Pitfalls

### Pitfall 1: Missing fetch-depth: 0 in GitHub Actions
**What goes wrong:** GoReleaser fails with "git tag not found" or generates empty changelogs.
**Why it happens:** `actions/checkout` defaults to shallow clone (depth=1), but GoReleaser needs full history for changelog generation and tag detection.
**How to avoid:** Always set `fetch-depth: 0` in the checkout step.
**Warning signs:** GoReleaser errors about missing tags or empty release notes.

### Pitfall 2: Cross-repo Token Permissions
**What goes wrong:** GoReleaser builds and releases successfully but fails to push the Homebrew formula to `revenium/homebrew-tap`.
**Why it happens:** The default `GITHUB_TOKEN` only has permissions for the repo where the workflow runs, not for external repos.
**How to avoid:** Create a PAT with `repo` scope, store it as a secret (e.g., `TAP_GITHUB_TOKEN`), and reference it in `.goreleaser.yml` as `brews.repository.token: "{{ .Env.TAP_GITHUB_TOKEN }}"`. Pass it in the workflow via `env: TAP_GITHUB_TOKEN: ${{ secrets.TAP_GITHUB_TOKEN }}`.
**Warning signs:** 403 or "resource not accessible by integration" errors during release.

### Pitfall 3: Completions Script Not Executable
**What goes wrong:** GoReleaser before hook fails because `scripts/completions.sh` is not executable.
**Why it happens:** File permissions not set after creating the script.
**How to avoid:** Run `chmod +x scripts/completions.sh` or use `sh scripts/completions.sh` in the before hook instead of `./scripts/completions.sh`.

### Pitfall 4: Archive Format Platform Mismatch
**What goes wrong:** Windows users can't extract `.tar.gz` archives easily.
**Why it happens:** Using a single archive format for all platforms.
**How to avoid:** Use GoReleaser's `format_overrides` to produce `.zip` for Windows and `.tar.gz` for Linux/macOS.

### Pitfall 5: Completion Files Not Included in Archives
**What goes wrong:** Completion files exist in `completions/` but don't appear in the release archives.
**Why it happens:** GoReleaser's `archives.files` section doesn't include the completions directory.
**How to avoid:** Add `completions/*` to the `archives.files` list so non-Homebrew users can also access completion scripts.

## Code Examples

### Complete .goreleaser.yml
```yaml
# Source: GoReleaser docs, goreleaser's own config, Makefile ldflags pattern
version: 2

before:
  hooks:
    - go mod tidy
    - sh scripts/completions.sh

builds:
  - main: .
    binary: revenium
    env:
      - CGO_ENABLED=0
    ldflags:
      - -s -w
      - -X github.com/revenium/revenium-cli/internal/build.Version={{.Version}}
      - -X github.com/revenium/revenium-cli/internal/build.Commit={{.Commit}}
      - -X github.com/revenium/revenium-cli/internal/build.Date={{.Date}}
    goos:
      - linux
      - darwin
      - windows
    goarch:
      - amd64
      - arm64

archives:
  - format: tar.gz
    format_overrides:
      - goos: windows
        format: zip
    files:
      - README*
      - LICENSE*
      - completions/*

changelog:
  sort: asc
  filters:
    exclude:
      - "^docs:"
      - "^test:"
      - "^chore:"

brews:
  - repository:
      owner: revenium
      name: homebrew-tap
      token: "{{ .Env.TAP_GITHUB_TOKEN }}"
    directory: Formula
    homepage: https://github.com/revenium/revenium-cli
    description: Manage your Revenium account from the command line
    license: MIT
    install: |-
      bin.install "revenium"
      bash_completion.install "completions/revenium.bash" => "revenium"
      zsh_completion.install "completions/revenium.zsh" => "_revenium"
      fish_completion.install "completions/revenium.fish"
    test: |
      system "#{bin}/revenium", "version"
```

## State of the Art

|--------------|------------------|--------------|--------|
| `brews` only | `brews` + `homebrew_casks` | GoReleaser v2.10 (June 2025) | `brews` was briefly deprecated, then un-deprecated in v2.12. For CLI tools, `brews` remains correct. |

## Open Questions

1. **License file**
   - What we know: The `.goreleaser.yml` `brews` section needs a license value and archive may include `LICENSE*`
   - What's unclear: Whether a LICENSE file exists in the repo
   - Recommendation: Check for LICENSE file; if missing, the planner should note it as a prerequisite but not block on it

2. **TAP_GITHUB_TOKEN secret**
   - What we know: A PAT with `repo` scope is needed to push to `revenium/homebrew-tap`
   - What's unclear: Whether the user has already created this token
   - Recommendation: Document this as a manual prerequisite in the plan. The user must create the PAT and add it as a GitHub Actions secret.
